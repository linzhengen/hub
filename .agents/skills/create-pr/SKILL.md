# hub PR 作成規範

## 目標

1. 現在のブランチとベースブランチの全変更内容に基づいてPRを生成し、最後のコミットだけでなく全体を考慮する。

2. リポジトリにあるテンプレートを厳格に使用し、独自のPR構成を勝手に作らない。

3. ユーザーの言語（日本語または英語）に応じてテンプレートを選択するが、PRのタイトルは常に英語とし、規定の命名形式に従う。

4. 実際に `gh pr create` を実行する前に、必ず `base`、`title`、`body` をユーザーに確認させ、承諾を得てからPRを作成する。

## トリガーシーン

以下のいずれかの場合に、このスキルを使用します：

- PRを作成、Pull Requestを開始
- PRのタイトルまたは説明を書く
- 現在のブランチの変更をPR用に要約する
- `gh pr create` を使用して `hub` にPRを出す

## 基本ルール

### 1. リポジトリのテンプレートを優先

常に以下のテンプレートのいずれかを使用します：

- 日本語：`.github/pull_request_template_ja.md`
- 英語：`.github/pull_request_template.md`

セクション名を勝手に変更したり、テンプレートの基本構造を削除したりしないでください。コメントやプレースホルダーは削除しても構いませんが、最終的に提出するセクションは維持してください。

### 2. 言語はユーザーに合わせるが、タイトルは英語固定

以下の優先順位で判断します：

1. ユーザーの現在のリクエストが主に日本語 -> 日本語テンプレートを使用
2. ユーザーの現在のリクエストが主に英語 -> 英語テンプレートを使用
3. 混合しているが、履歴が主に日本語 -> 日本語テンプレートを使用
4. 判断できない場合はユーザーに確認する（勝手に推測しない）

コードやブランチ名、コミットメッセージが英語だからといって、無理に英語テンプレートを使用する必要はありません。

ただし、テンプレートが日本語・英語のいずれであっても：

- `PR title` は必ず英語にする
- `PR title` は規定のフォーマット（type/scope等）に従う
- `PR body` の内容は選択した言語に合わせる

### 3. ブランチを分析してからPRを書く

PRを作成する前に、以下を必ず確認してください：

- 現在のブランチ名
- ベースブランチ
- ベースブランチに対する現在のブランチの全コミットリスト
- `base...HEAD` の完全な diff

作業ディレクトリの未コミット内容や、最新の1つのコミットだけで判断しないでください。

### 4. まず下書きを提示し、その後にPRを作成

ユーザーが「直接PRを作成して」と言った場合でも、以下の手順を踏みます：

1. `base`、`title`、`body` の下書きを生成
2. ユーザーに対し、これが提出予定のPR内容であることを明確に伝える
3. 作成を継続するか、修正が必要かを確認する
4. ユーザーの明確な同意を得てから、初めて `gh pr create` を実行する

ユーザーが途中でタイトルやタイプ、チェンジログ、ターゲットブランチの変更を求めた場合は、下書きを更新して再度確認してください。

### 5. タイトルと本文の役割分担

- PRタイトル：ブランチの主要な変更内容を英語1文で要約
- PR本文：背景、解決方法、関連Issue、更新ログへの影響を説明

本文はファイルごとの修正リストではありません。「なぜ変更したか」「開発者やユーザーにどのような影響があるか」を要約してください。

### 6. 情報不足な場合は無理に書かない

以下が不明で、ブランチの内容からも確実に推測できない場合：

- ベースブランチ
- 関連Issue
- 変更の性質（type）
- テストまたは検証方法

下書きでは不明な点を明示し、ユーザーに確認を求めてください。

## 実行ステップ

### 1. リポジトリとPR環境の確認

以下を確認することを推奨します：

```bash
git status --short
git branch --show-current
git remote -v
gh auth status
```

`gh` が使えない、ログインしていない、gitリポジトリでない、現在のブランチがPRに適さない場合は、問題を説明し、処理を継続しないでください。

### 2. ベースブランチの特定

デフォルトで `main` を使うのではなく、以下の順で判断します：

1. ユーザーが `base branch` を明示している場合 -> それを使用
2. ブランチの Git 情報から推測する：
   - `git branch -vv` で tracking/upstream を確認
   - `git reflog show <current-branch>` でチェックアウト元を確認
   - 必要に応じて `git merge-base HEAD <candidate-branch>` で分岐点を比較
3. あるブランチから派生したことが確実な場合は、そのブランチを `base` にする
4. 判断できない場合は、デフォルトのブランチ（通常は `main`）とする

推奨コマンド：

```bash
git branch --show-current
git branch -vv
git reflog show --date=local $(git branch --show-current)
git remote -v
```

### 3. 全変更内容の収集

少なくとも以下を確認してください：

```bash
git log --oneline <base>..HEAD
git diff --stat <base>...HEAD
git diff <base>...HEAD
```

PRに含まれる全コミットを網羅し、最後の変更だけに偏らないように要約します。

### 4. テンプレート言語の判断

「ユーザーの習慣に合わせる」ルールに基づき選択：

- 日本語テンプレート：`.github/pull_request_template_ja.md`
- 英語テンプレート：`.github/pull_request_template.md`

該当するテンプレートを読み込んでから記入してください。

### 5. PRタイプの判断

変更の「主目的」に基づいて判断します。

優先判断順位：

1. サーバー側のロジック、API、ドメインロジックの変更
   - `server` 関連、`fix` または `feat`
2. UI、Reactコンポーネント、フロントエンドの変更
   - `ui` 関連、`fix` または `feat`
3. インフラ、CI/CD、Kubernetes、Terraform の変更
   - `infra` / `ci` / `chore`
4. ドキュメント、説明文の修正
   - `docs`
5. リファクタリング、テスト、パフォーマンス改善
   - `refactor` / `test` / `perf`

プロジェクトの `commitlint.config.js` は `@commitlint/config-conventional` に従っています。

### 6. PR核心情報の要約

少なくとも以下を整理します：

- Background and Solution：問題の背景、解決方法、外部的な影響
- Change Log：ユーザー感知可能な影響を記述（内部的な詳細は不要）
- Related Issues：Issue へのリンクまたは `close #xxxx` 等
- Checklist：`make test`, `make gen` 等の実行確認

### 7. Change Log の扱い

以下のようなケースでは、実質的な更新ログは不要です：

- `docs`
- `ci` / `chore`
- テストのみ
- 内部的なリファクタリングでユーザー感知がないもの

これらの場合：
- 無理にログを捏造しない
- テンプレートのセクションは残す
- `N/A` や `No changelog required` と記載する

### 8. PRタイトルの生成

タイトル要件：
- 英語で記述
- `type(scope): subject` 形式（例: `feat(server): add user api`）
- コミットメッセージの単なるコピーではなく、ブランチ全体の目的を反映させる

### 9. PR本文の下書き作成

記入時の注意：
- テンプレートのコメントや説明文は削除
- セクションタイトルは維持
- チェックリストは、変更内容から推測できる範囲で埋める（不明な点は空欄）

### 10. ユーザーへの確認

出力には以下を含めてください：
- `Base branch`
- `PR title`
- `PR body`
- ユーザーに補足や確認を求める点

明確に確認を得るまで `gh pr create` は実行しません。

### 11. PR作成

確認が得られたら実行します。

```bash
gh repo view --json nameWithOwner
```

リポジトリ名を確認し、正しいリポジトリにPRを作成するようにしてください。

実行例：

```bash
gh pr create --base <base> --title "<title>" --body "$(cat <<'EOF'
<body>
EOF
)"
```

作成成功後、PRのリンクを表示します。

## 書き方要件

### タイトル (Title)

- 英語固定
- 形式: `type: subject` または `type(scope): subject`
- subject は動詞の原型から始め、結果を簡潔に書く
- `update`, `fix issues`, `misc changes` 等の抽象的な表現は避ける

### Background and Solution

- 何を解決したいか、どのような方法を採用したか
- API や UI 等の変更点を明示

### Change Log

- ユーザー感知可能な変更がある場合のみ記述
- 不要な場合は `N/A` 等を使用
