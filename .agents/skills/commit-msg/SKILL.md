# hub コミットメッセージ生成規範

## 目標

1. **変更内容を正確に要約** - 現在の **git 暫定領域 (staged)** の内容に基づき、今回の変更を網羅する1行の英語コミットメッセージを生成する。

2. **プロジェクト規約の遵守** - `hub` で採用されている Conventional Commits 規約 (`type(scope): subject`) に厳格に従う。

## 核心原則

> コミットメッセージは単なる diff のファイル羅列ではなく、今回の変更の「意図」を凝縮したものであるべきです。暫定領域を読み、要約し、英語で1行出力します。

## トリガーシーン

以下のいずれかの場合に、このスキルを使用します：

- コミットメッセージを書いてほしい、作成してほしい
- 「msg」「commit msg」などの入力があり、文脈が git コミットである
- 現在の変更内容（staged）から1行の要約を生成してほしい
- 暫定領域の内容を要約して、コミット用のタイトルを作ってほしい

## 基本ルール

### 1. 暫定領域 (Staged changes) を優先

デフォルトでは **staged changes**（`git diff --cached`）の内容のみに基づいてメッセージを生成します。実際にコミットされるのは暫定領域の内容だからです。

ユーザーが明示的に「未暫定の変更も含めて」と言った場合のみ、`git diff`（unstaged）も参照します。

### 2. メッセージは英語、解説はユーザーの言語

生成されるコミットメッセージ自体は **常に英語** で出力します。
ただし、それ以外の解説や確認メッセージは、ユーザーの言語（日本語）に合わせてください。

### 3. デフォルトで最終的な1行のみ出力

ユーザーから「理由を説明して」などの指示がない限り、最終的な回答は **コミットメッセージ1行のみ** とし、分析、箇条書き、コードブロック、引用符などは含めないでください。そのまま `git commit -m "..."` で使える形式を優先します。

## 実行ステップ

### 1. git 状態と変更内容の読み取り

推測で書かず、必ず以下のコマンドを実行して事実を確認してください。

```bash
git status --short
git diff --cached --stat
git diff --cached
git log --oneline -10
```

- `git status --short`: どのファイルが暫定（staged）されているか確認。
- `git diff --cached --stat`: 変更の規模と範囲を把握。
- `git diff --cached`: 実際の変更内容を詳細に確認。これがメッセージ生成の根拠になります。
- `git log --oneline -10`: プロジェクトの最近のコミットスタイル（特に scope の使い方）を確認。

### 2. 生成可能かの判断

暫定領域が空の場合：
- メッセージを捏造しないでください。
- 暫定された変更（staged changes）がないため、メッセージを生成できない旨をユーザーに伝えてください。

### 3. 変更の主目的の特定

`git diff --cached` の結果から：
1. **主要な目的** を特定：新機能、バグ修正、ドキュメント、リファクタリング、テスト、CI/CD、依存関係の更新など。
2. **影響範囲 (scope)** を特定：`server`, `ui`, `infra`, `pkg`, `proto` など。
3. 複数の変更が含まれる場合、それらを包含する1つ上の抽象概念で要約し、長い文章にならないようにします。

### 4. Conventional Commits への適合

`hub` の規約に従います。

```text
<type>(<scope>): <subject>
```

または scope がない場合：

```text
<type>: <subject>
```

### 5. 1行メッセージの生成要件

- **1行であること**
- **全ての暫定内容をカバーしていること**
- **簡潔であること**
- **英語であること**
- **そのままコミットに使用できること**

## 書き方ルール (Conventional Commits)

### type の選択

`commitlint.config.js` の設定に基づき、以下から選択します：

- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響を与えない変更（ホワイトスペース、フォーマット、セミコロンの欠落など）
- `refactor`: バグ修正も機能追加も行わないコード変更
- `perf`: パフォーマンスを向上させるコード変更
- `test`: 不足しているテストの追加や既存のテストの修正
- `build`: ビルドシステムや外部依存関係に影響を与える変更 (例: `go.mod`, `Makefile`)
- `ci`: CIの設定ファイルやスクリプトへの変更 (例: GitHub Actions)
- `chore`: その他の変更 (ソースやテストファイルの変更を含まないもの)
- `revert`: 以前のコミットを元に戻す

### scope の選択

`hub` プロジェクトの構造に基づいて選択します：

- `server`: Go バックエンド関連
- `ui`: フロントエンド（React）関連
- `infra`: Terraform, Kubernetes, CI/CD 関連
- `proto`: Protocol Buffers 定義関連
- `db`: データベース、マイグレーション関連
- `pkg`: 共通パッケージ関連

複数のディレクトリにまたがる場合は、最も重要なものを選択するか、広域な名称を使用します。

### subject のルール

- 英語で記述
- 祈使形（Imperative mood）を使用（例: `add` ではなく `added` や `adds` は使わない）
- 1文字目を大写しにしない
- 最後にピリオド (.) を置かない
- 50文字以内（最大72文字）を推奨

## 禁止事項

- `git diff --cached` を読まずにメッセージを生成すること
- ファイル名だけで内容を推測すること
- 一部の変更だけを記述し、他の重要な変更を無視すること
- 複数行の出力をすること（ユーザーからの要求がない限り）
- `WIP` や `update files` などの抽象的で意味のないメッセージを書くこと
